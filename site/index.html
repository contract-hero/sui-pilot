How we're learning to feed Claude the Sui docs

sui-pilot is a doc-first plugin: it ships the current Sui, Move, Walrus, Seal and TypeScript-SDK docs so Claude reads them instead of trusting stale training memory. The hard question was never which docs — it's how they should enter the context window. This is the running log of how we measure that — two evals so far, plus a lab to see why.

The story in two chapters — one question: how should the docs reach Claude?

  1. 1 Chapter 1 — does the borrowed machinery earn its keep?

    Pipe-delimited index vs. the full plugin copycat

    sui-pilot started by cloning a heavyweight approach: a Vercel-style plugin with a borrowed matcher pipeline and a fat pipe-delimited doc index baked into the preamble. The first eval A/B'd that machinery against leaner shapes across a 27-task suite — Move 2024 syntax, one-time-witness, SDK 2.0, stale-training traps, multi-file refactors, ambiguous specs — scored on literal pass/fail, an LLM rubric, token economics and an optional compile gate. After a few baseline runs the answer settled: none of the borrowed machinery beat a slim table that just points at the docs. We deleted ~17 K LOC and kept the pointer — plus the eval harness, which became the instrument chapter 2 runs on.

  2. 2 Chapter 2 — point at the docs, or map them?

    Pointing vs. docs-section mapping

    Chapter 1 left us pointing at the docs. Chapter 2 tests the one piece of the old plugin that might still earn its place: a mapping of the docs' sections — a 445-line relational ecosystem graph (abilities, object model, auth patterns and how they connect) loaded straight into the prompt. So the axis is: a bare pointer (the 7-row routing table, searched on demand) versus that map always present. Same harness as chapter 1, a fresh 4-task NFT-dapp fixture, byte-identical everything except that one file. The two shapes performed differently — and where the measurement got murky turned out to be the most useful part.

Side tool — not a chapter

The context-window lab

Neither eval explains, on its own, why a shape behaves the way it does. For that we keep an interactive lab that walks what actually enters Claude's head during a real Sui workflow — bundled docs, MCP calls, skill payloads, the agent file, step by step. It isn't a chapter in the story; it's the bench instrument we read the story against.

What's still open

Chapter 2 is a checkpoint, not a verdict. Code quality came out indistinguishable between pointing and mapping, and wall-time and cache-reads dropped with the always-loaded map — but the cost claim isn't trustworthy yet, because the map nudged the agent into dispatching background subagents whose tokens our harness doesn't count. Until tokens.csv walks the full subagent tree, "cheaper" stays unproven.

Next, we'll point the lab at the same question: capture the context window side by side when the agent merely points at the docs versus when the section-map is loaded, so we can see the difference rather than infer it. The recurring lesson holds — measuring what context does is harder than changing it, and the methodology keeps turning out to be part of the result.
