Formal verification for Move — the Sui Prover investigation.

How sui-pilot grew a formal-verification capability: a /specify skill that authors #[spec(prove)] specifications, a /verify skill that re-checks them against drifting code, an MCP server wrapping the sui-prover binary, and a bundled prover-docs corpus — hardened by a live evaluation against a production AMM. This page is the candid record: what shipped, what the eval exposed, and what is still open.

Why formal verification, and what the Sui Prover is

Tests sample behavior; a prover exhausts it. For Move packages guarding real value, a machine-checked proof that a property holds on every input is a different class of assurance than a green test suite.

The Sui Prover (by Asymptotic) verifies Sui Move code against specifications written as ordinary Move functions annotated with #[spec(prove)], using requires, ensures, asserts, and macros like forall!, exists!, and invariant!. It installs via brew install asymptotic-code/sui-prover/sui-prover and expects implicit framework dependencies in Move.toml.

Crucially, this is not the legacy Move Prover's MSL. The old syntax an LLM remembers from training — aborts_if, pragma, apply, assume, free axiom — is not what the Sui Prover uses. That is exactly the failure mode sui-pilot exists to fix, so the plugin bundles .sui-prover-docs/ as its sixth doc corpus (20 files): guide/ with the canonical installation and spec-reference prose, sources/ with the actual prover construct definitions specs import (prover.move with native requires/ensures/asserts plus Boogie tuning hooks, ghost.move for ghost global state), and examples/ with four working spec packages from asymptotic-code/sui-kit.

“FV is only as good as the specs you can write — so the deliverable has to make spec intent and coverage auditable, not just hand the prover machine-readable clauses.”
Design premise of the deliverable redesign, May 2026

What ships today

One MCP server, one bundled docs corpus, and two skills with an explicit boundary between them: /specify authors; /verify checks.

Skill

/specify — the producer

An interactive, multi-phase orchestrator that authors #[spec(prove)] specs for every externally reachable function (public non-package plus entry) of a Sui Move package — validate, discover, cascade, prioritize, then a per-function draft → prove → iterate loop, ending in an HTML report and a reproducibility manifest.

Skill

/verify — the consumer

Read-mostly: it never authors, edits, or deletes a spec. It reads the manifest, detects drift, re-runs the prover on every spec package with its recorded flags, and reports which guarantees still hold. Uncovered functions get routed back to /specify.

MCP server

sui-prover-mcp

A TypeScript wrapper around the sui-prover binary exposing exactly three tools: prove_package, list_specs, prover_capabilities. By design it never edits the user's Move.toml — setup issues surface as setup_warning findings.

The load-bearing decision: specs live in a sibling package

Verification scaffolding never touches deployed source. Specs land in a separate <pkg>_specs/ package of #[spec(prove, target = ...)] twin functions — one <mod>_specs.move per source module — leaving the production package with no #[spec_only], no prover dependency, no marker blocks. Inline emission remains available as an --inline opt-out. Functions that need a non-default prover invocation (e.g. --no-bv-int-encoding for bit-exact semantics) move to a lazily-created second <pkg>_specs_bv/ package with its own recorded flag set.

MCP tool surface

ToolWhat it doesNotable design choices
prove_package Spawns the sui-prover binary shell-free with --path, --timeout (default 60s per spec), optional function/module scoping, and an extra_args escape hatch. Returns a typed summary.overall (verified_all / failed_some / no_specs / compile_failure / timeout / error) plus findings tagged with a 19-kind taxonomy; unknown extra flags are dropped with a warning, never forwarded; raw output is ANSI-stripped and truncated on compile failures.
list_specs Comment-aware regex scanner for #[spec(...)]-annotated functions across a package. Exists because the binary has no “list existing specs” command; used by /specify for idempotency and resume.
prover_capabilities Probes binary presence/version/flags, cloud config, Sui toolchain version, per-package setup warnings, and declared git dependencies. Gates on Sui 1.45+ implicit framework deps; reachability probing of git deps is deliberately left to the caller.

Two details worth knowing

Discovery is regex, not LSP: an empirical probe showed move_document_symbols collapses all visibilities to kind: "function", so Phase 1 runs comment-aware regex over the source to find the public/entry surface. And stub or external callees are axiomatized with #[spec(skip, target = ...)] functions in a specify_axioms.move sidecar — every such axiom is trusted, not proved, and each one is disclosed in the report's “Trusted axioms — not proved” section.

The producer/consumer split

A proof is only meaningful while it is about the current code. The split exists to make that binding explicit and checkable.

“A spec is a claim of the form ‘this code, at this state, against these deps, satisfies this property.’ /verify's job is to answer ‘is that claim still true, and is it still about the current code?’”
skills/verify/SKILL.md
Binding

spec-context.json

/specify ends every run by writing a reproducibility manifest: per-function source hashes keyed <mod>::<fn>, spec packages with their prover flags, dependency pins, and toolchain versions (sui-prover, Sui CLI, Boogie, Z3).

The default hash recipe (normalized_source) strips comments, collapses whitespace, then takes the first 16 hex chars of sha2-256 — so reformatting does not register as drift, but any token change does.

Checking

Three drift axes, six verdicts

/verify detects source drift (recompute each hash with the exact recipe /specify used), coverage drift (re-run discovery; find uncovered new functions and orphaned specs), and context drift (toolchain versions and git-dep revs vs the manifest).

It then joins prover result with hash freshness into verdict buckets: holds, violated (fresh + failed = real regression), stale-passing, stale-failing, uncovered, orphaned. A violated verdict means you broke a property you committed to; an uncovered one means you entered territory the specs never described.

CI posture

Default mode is warn-and-run: non-zero exit only on hard spec failure. /verify --strict is the CI contract — refuse-on-drift, non-zero exit on any spec failure, staleness, uncovered function, missing spec package, or context drift. It is exactly what the prover.yml stub emitted by /specify --audit runs, because CI must catch “the proof is no longer about the current code,” not just “a spec failed.” Without a manifest, /verify degrades gracefully: it re-proves the spec packages but announces that drift detection is unavailable.

Empirical grounding: the Aftermath AMM eval

On 2026-05-14 — the day the stack landed — a live /specify run was observed against a real production codebase: Aftermath Finance's public AMM.

The target: 4 packages, 229 externally reachable public/entry functions, private git dependencies, and a stub-only utilities::fixed module whose every function body is abort 0. The method: a second Claude Code session in a sibling worktree armed a tail -F on the target session's transcript and a 5-second mtime poll on its files — non-intrusive observation, no edits to the target repo. Toolchain: sui-prover 1.5.3, sui 1.69.2. Verdict: Conditional Pass

Outcome

47 verified specs

100% spec coverage of amm-math's externally reachable surface: 47 verified specs plus 22 callee axioms in a sidecar — 141 verification subchecks, 0 failures, 0 timeouts, on a real AMM math kernel covering weighted geometric mean and the Curve stable curve.

Caveat

ignore_abort everywhere

Every spec used ignore_abort — in the report's own words, “postcondition smoke tests, not full abort-safety proofs.” A follow-up pass removing it and adding precise abort conditions was flagged as the next step.

Findings

12 fixes, same day

3 BUGs, 6 GAPs, 3 UX items (plus 3 WINs) — all 12 fixed in PR #27 within hours: output-parser rewrite, summary.overall, structured failure kinds, a hard Phase 0 compile gate, a scope + callee-quality probe, and taxonomy docs.

The most critical finding

BUG-03: the MCP failed to parse successful prover output. A run with four checkmark verifications and a literal “Verification successful” line still returned summary {verified: 0, failed: 0, skipped: 0, timeouts: 0} and findings: [].

“Any agent or eval rig that trusts the summary will conclude the run did nothing — the exact opposite of the truth. … This is the single highest-leverage fix in the sui-prover-mcp surface — every other finding in this report becomes secondary if the structured output is wrong.”
evals/specify-aftermath-eval/report.html, BUG-03

The report's conclusion: the feature is conclusively workable on hard real-world Move code — the GAP/BUG/UX items were “friction in the surrounding tooling, not capability gaps.” The eval also caught things the skill spec never taught: the agent spotted the stub-callee vacuous-verification trap unprompted, and the blocked-state artifacts (a structured .specify-progress.json blocker record plus a self-contained HTML report) exceeded the documented contract.

This live-observation eval is distinct from the project's automated v1-vs-v2 A/B eval framework (27 tasks as documented, since grown to 32 in evals/tasks.json) — it was a one-off production-grade probe of the /specify deliverable itself, not a plugin-version comparison. The full write-up lives in the repo at evals/specify-aftermath-eval/report.html.

The deliverable redesign: five locked decisions

The 2026-05-19 design exploration resolved five questions about what a formal-verification deliverable should be. Four of the five decisions (Q1, Q2, Q4, Q5) are verifiably present in today's skills/specify/SKILL.md; Q3's source-rendering rule is not yet reflected in the skill text.

QuestionLocked decision
Q1 · Granularity Hybrid — per-module reports next to source (<mod>.spec.html) plus a package-level index.html for cross-module invariants; --single-file opt-in for tiny packages.
Q2 · Living vs. snapshot Structured sections via HTML comment markers — specify:auto:* zones regenerate every run, specify:human:* zones are preserved — on a strong opinionated template.
Q3 · Source rendering Hybrid — function signature + spec body inline, link out for full source; --audit upgrades to full-source-inline. not yet in SKILL.md
Q4 · Context binding v1 = manifest only (spec-context.json capturing source hashes, dep commits, toolchain versions). v2 = preflight dep clone. v3 = drift enforcement.
Q5 · Narrative structure Invariant-driven primary view + function-index appendix. Invariants get a human name and a stable slug (INV-supply-conservation), linked from source via explicit // invariant: markers, each with Threat model and Proof obligations sections.

The implementation (PR #30) also folded in nine learnings from a real Sui-Prover engagement — the integer-library work from Asymptotic × Bluefin — including abort conditions as first-class output, math-domain modeling for custom integer types, defining-property postconditions, per-package prover flags, and --audit regression artifacts with a prover.yml CI stub. Q4's v1 shipped exactly as designed; v3's drift enforcement became /verify --strict.

Not done yet

The design was never walked back through the Aftermath AMM eval to check whether the 12 findings would have been caught by the redesigned deliverable — no commit evidences that validation. The eval's own recommended follow-up — a /specify run on a fully-buildable package to exercise the happy path of Phases 2–4 in production — also remains open. And the Aftermath specs' ignore_abort caveat still stands.

Known doc drift: commands/specify.md still describes the legacy inline-marker layout as the default, while skills/specify/SKILL.md has moved the default to the sibling-package layout with --inline as the opt-out. The command doc lags the skill. Smaller notebook noise: the wave-2 design note is still an untracked file in the working tree, and it dates the AMM eval as “yesterday’s” on 2026-05-19 even though the fix commit landed 2026-05-14 — the record keeps the tension rather than smoothing it over.
