sui-pilot doc-fetching — behavioral investigation notes

Captured: 2026-05-19  ·  Source run: evals/results/2026-05-19T12-20-07Z/  ·  Purpose: seed observations for the next iteration on sui-pilot's doc-discovery layer.

This is a notes document. It is deliberately not a verdict and does not validate or invalidate any prior hypothesis about which doc-discovery shape is "best". It records the behavioral deltas we actually saw between two configurations of the agent prompt and the methodology issues that surfaced, so the next iteration can start from concrete signal instead of starting over.

What was run, very briefly

A 4-task NFT-app build (Move package + Vite/React/dApp Kit frontend) was issued to two configurations of the sui-pilot-agent system prompt:

  • Slim only — the 7-row routing table that points at the bundled .<source>-docs/ corpora.
  • Slim + always-loaded ecosystem graph — the same routing table with the 445-line relational knowledge graph (recovered from the Vercel-port-era sui.md) appended to the agent prompt so the cross-link map is always present.

Bundled corpora, skills, MCP servers, fixtures, model, and prompts were byte-identical across the two configurations. The only difference was the agent-file content. The Sui NFT minting bootcamp (~/workspace/sui-move-bootcamp/) was used as a hidden reference for grading; the per-task fixtures were empty scaffolds that the model built into a working dapp.

Behavioral observations

obs-1 Subagent-dispatch likelihood diverges

With the always-loaded ecosystem graph present, the agent dispatched background subagents for work on at least two of four tasks (visible in the parent transcripts as "the earlier background search has completed" / "acknowledged — that was the background find from earlier"). The slim-only configuration tended to keep the work in-process. We don't yet know what in the prompt context tips the decision toward delegation, but the cross-link map (with its many 📖 docs: pointers) is a plausible trigger.

obs-2 Parent-level output tokens are no longer a usage proxy when subagents are in play

Task-level output tokens swing by two orders of magnitude on the same prompt. Example:

TaskSlim parent outputGraph parent output
task-nft-01-move-package27,08134
task-nft-02-dapp-kit-setup1778,186
task-nft-03-mint-and-list-components11,655248
task-nft-04-full-stack-smoke30,43425,944

The 34/248/177 entries are runs where the work happened in a subagent; the parent only emitted a closing ack. The usage JSON block we currently aggregate is the parent's only. Conclusion: any future cost or productivity claim made from tokens.csv as-is is partially fiction. Task-04 is the most representative single cell because both arms kept the work in-process.

Wall time is partly confounded by obs-1. The task-04 cache-read delta is the cleanest single number — the always-loaded graph appears to shave some opportunistic Glob/Grep loops the slim arm would otherwise issue, at the cost of more static input. Sample size is one task; do not over-fit.

obs-4 Codegen quality is indistinguishable on the three solvable tasks

Spot-checking diffs: same module shape, same Move 2024 idioms, same dApp Kit hooks, same four-moveCall PTB with the hero handle threaded correctly, same React Query / getObject pattern for the shared registry, same SDK 2.0 / gRPC client choice, no SDK 1.x leakage in either arm. The behavioral difference is in how the answer is produced, not what the answer is.

obs-5 The same prompt at different points in a session can take wildly different time

The smoke run of task-nft-02 (single task, single config) took ~3 minutes; the same task inside the full multi-task run took ~52 minutes. Same prompt, same fixture, same agent file. The agent's strategy is sensitive to whatever else is in its working state (cache priming order, MCP/skill load order, doc files already-read in-session). Single-task wall-clock samples are noisy.

obs-6 Cache-hit ratios are already very high in both configurations

96.8% and 97.4%. The bulk of context loaded across the run is being cached effectively. Differences between configurations are in the volume of context loaded (~28% delta in cache reads) more than in how often that context hits the cache.

Methodology issues that need cleaning up before any rerun

meth-1 Subagent token rollups

The harness aggregates .usage from the parent claude -p response only. When the agent dispatches subagents (Phase F-style fanout, background search), those subagents' input/output tokens never appear in tokens.csv. The aggregator needs to walk the JSON tree (or the per-task transcript) for nested usage blocks before token-economics claims hold.

meth-2 Diff pollution from .remember SessionStart hooks

Every per-task tmpdir collected a .remember/ directory written by the remember plugin's SessionStart hook (logs and a .gitignore). These show up as ~30–50 line additions in every .diff. The literal pass criteria still pass, but the diff size becomes a poor proxy for "what the model actually wrote". Two options: (a) instruct the runner to delete .remember/ before diffing, or (b) git diff ---style filter the diff to .move/.ts /.tsx/package.json/Move.toml when scoring.

meth-3 compileAfter only knows the tmpdir root

task-nft-04 ships a Move package under $tmpdir/move/hero/. The harness invokes sui move build at $tmpdir and fails uniformly because there is no Move.toml at the root. We added tsBuildSubdir for the TS pipeline already; mirror it with compileSubdir for Move, then re-run task-04. Without that, the rubric on task-04 never gets exercised because the literal compile gate fires first.

meth-4 Path-case sensitivity in passCriteria.additionalFiles

Tasks expected src/dApp-kit.ts; both arms wrote src/dapp-kit.ts. On macOS APFS these are the same file; in the diff they are not the same path string. Either normalize comparisons or rewrite criteria to use lowercased / glob paths.

meth-5 Scorer prompt label drift

compare-prompt.md still carries preamble text describing the original matcher-pipeline vs slim comparison; the auto-scored score.html from this run swapped the v1/v2 labels in its prose as a result. The numbers were fine. Parameterizing the labels per run (passing arm descriptors as placeholders the way RESULTS_DIR already is) would prevent that.

Open questions for next iterations

  • q1 What in the prompt context tips the agent toward subagent dispatch? Is it the cross-link density of the ecosystem graph, the additional surface area for delegation, or something else? Cheap experiment: ablate the graph down to just the Legend + one section and see if delegation behavior persists.
  • q2 If the graph's value is mostly cost reduction at parity quality, is there a subset that captures most of the win at a fraction of the always-loaded cost? Worth running with the graph trimmed to the Legend + ABILITIES + Sui object-model section only.
  • q3 Are the 📖 docs: pointers in the graph actually followed? Instrument the transcript scanner to count tool calls whose paths match pointers that appear in the prompt context vs. ones discovered ad-hoc; that separates "graph is used as a TOC" from "graph is read and ignored".
  • q4 Is the same wall-time / cache-read delta present on task categories that aren't NFT-flavored — multi-file refactors, stale-training-API tasks (task-02-sdk-2-client, etc.), token-pressure tasks (task-27-long-prompt-spec)? The existing 28-task suite was not re-run in this batch; doing so would generalize the signal.
  • q5 Does always-loading the graph change behavior in negative ways for short, mechanical tasks (typo fixes, single-line edits) where the extra context is pure overhead? Quick check: run the smallest existing tasks (task-01-module-syntax, task-04-vector-method-syntax) with the graph present and compare token economics.
  • q6 Background subagents are eating cost that doesn't surface in tokens.csv. After meth-1 is fixed, re-score the same captured transcripts (no re-run cost) and check whether the cost-reduction story holds when total billed tokens are honest.
