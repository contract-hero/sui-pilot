sui-pilot · experiment 3

Point at the docs, or index them into the prompt?

sui-pilot ships the live Sui, Move, Walrus, Seal and TS-SDK docs so Claude doesn't lean on stale training memory. But shipping the docs is only half the question. The other half: how should that knowledge enter the context window? This is what happened when we ran the two obvious answers head-to-head.

Source run: evals/results/2026-05-19T12-20-07Z/  ·  Build: 4-task NFT dapp (Move package + Vite/React/dApp Kit)  ·  Reference held hidden during grading.

The two ways to inject

Everything downstream of the agent was held constant — bundled corpora, skills, MCP servers, fixtures, model, and the task prompts were byte-identical. The only thing that changed between the two arms was the content of the agent's system file:

Arm A — point

A 7-row routing table

The slim agent file: a small table that says "Move syntax lives in .move-book-docs/, Sui runtime in .sui-docs/…" and tells the agent to Glob/Grep the right corpus on demand. The knowledge is reachable, not present. The agent decides when to go look.

Arm B — index

A 445-line ecosystem graph, always loaded

The same routing table, plus a relational knowledge graph appended to the prompt: abilities, object-model ownership, authorization patterns, the off-chain stack — with cross-links and 📖 docs: pointers. The map of how the pieces relate is always present, before the agent reads a single file.

What we expected vs. what we got

The bet was simple: indexing the relational map costs static tokens up front but saves the agent from flailing around the corpus, so it should be faster and maybe produce better-connected code. Quality was the first thing we checked — and it's where the experiment got interesting by refusing to move.

Code quality: indistinguishable

Spot-checking the diffs across the three solvable tasks: same module shape, same Move 2024 idioms, same dApp Kit hooks, the same four-moveCall PTB with the hero handle threaded correctly, the same React-Query / getObject pattern for the shared registry, the same SDK 2.0 / gRPC client choice, and no SDK 1.x leakage in either arm. The difference between the two shapes is in how the answer gets produced, not what the answer is.

Where it got murky — and why that's the real finding

The wall-time number above is too good to take at face value, and chasing down why is what this experiment actually taught us. With the ecosystem graph in context, the agent started dispatching background subagents for whole tasks — visible in the transcripts as "the earlier background search has completed." The slim arm tended to keep the work in-process.

That single behavioral shift quietly broke our measurement. Per-task parent output tokens swung by two orders of magnitude on the same prompt, depending on whether the work happened in the parent or in a subagent:

The honest caveat. Any cost-per-task claim drawn from tokens.csv as it stands today is partially fiction. The subagents that the graph encourages do real, billed work that our token accounting silently drops on the floor. "The graph is cheaper" is not a claim this run can support — only "the graph is faster on the wall clock and reads less cache," with a caveat that wall time is itself confounded by the delegation behavior.

The takeaway

Quality is shape-neutral. Pointing and indexing produced the same code. If you only care about correctness on these tasks, either shape works.

Indexing changes behavior, not just speed. The always-loaded relational map nudges the agent toward delegation and fewer ad-hoc corpus searches. That's a genuinely different operating mode, and it's the lever worth understanding next.

The methodology is part of the result. The most valuable output of this run wasn't a winner — it was discovering that our instrument can't yet measure the thing the winner would be decided on. Measuring what context does is harder than changing what context is.

What the next iteration has to settle

  • Honest token accounting first. The aggregator must walk the subagent tree for nested usage blocks. Re-scoring the same captured transcripts (no rerun cost) then tells us whether any cost story survives once subagent tokens are counted.
  • What tips the agent into delegating? Is it the cross-link density of the graph, the extra surface area for subtasks, or something else? Ablate the graph to the Legend plus one section and see if the delegation behavior persists.
  • Is a smaller index most of the win? If the value is mostly latency at quality parity, a trimmed graph (Legend + abilities + object-model) may capture the bulk of it at a fraction of the always-loaded cost.
  • Does it generalize past NFT-flavored tasks? Re-run the multi-file refactor, stale-API, and token-pressure tasks under both shapes before claiming the latency delta is real.
  • Does always-loading hurt small tasks? For typo fixes and single-line edits, the extra 445 lines may be pure overhead — worth checking the smallest tasks explicitly.
