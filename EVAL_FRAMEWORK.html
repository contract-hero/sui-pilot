Evaluating Claude Code context for Sui Move development

A doc-first plugin claims to make Claude Code better at writing Sui Move code than the bare model. That's an empirical claim — and most teams ship plugins without measuring it. This is a write-up of one attempt to measure it honestly, including the parts where the answer wasn't what we hoped, and the eval framework that fell out of the work.

Subject: sui-pilot, a Claude Code plugin for Sui/Move development.  |  Audience: anyone evaluating whether — and how — to enrich an agentic-coding tool's context for a fast-moving ecosystem like Sui.  |  Companion repo: NOTES.md for the in-repo narrative, evals/BASELINE.md for raw data.

1. Why measure this at all

Training drift on Sui and Move is real and fast. The Sui framework, the Move standard library, and the @mysten/* TypeScript SDK ship breaking changes monthly. A model with a January-2026 cutoff will confidently generate code patterns that no longer compile, deprecated APIs that have been removed, and signatures that have changed shape.

Plugins like sui-pilot exist to fix this. The pitch is: "ground the agent in current bundled docs so it doesn't generate from stale training memory". That's a falsifiable empirical claim, not a self-evident one. You can imagine plausible reasons it might not deliver:

  • The model might already know the right answer for common patterns and not need to read docs.
  • The agent might not actually reach for the bundled docs even when they're available.
  • The added context might crowd out the model's reasoning budget on simple tasks.
  • The plugin's curation choices (which docs, in what form, with what routing) might be wrong.

Without measurement, all of these stay as untestable beliefs. With measurement, they become questions you can answer with a fixture suite and a couple of hours of compute.

Why this matters beyond sui-pilot: the same framework that measures sui-pilot's plugin shape can measure any Claude Code customization for Sui — a competing plugin, a different prompt strategy in a skill, a new MCP integration, a model upgrade, a fresh batch of bundled docs. The framework is the durable artifact; sui-pilot is the first thing it was pointed at.

2. What sui-pilot is

sui-pilot is a Claude Code plugin that bundles documentation for the Sui ecosystem and exposes development tooling:

  • ~695 doc files across five corpora (.sui-docs/, .move-book-docs/, .walrus-docs/, .seal-docs/, .ts-sdk-docs/) bundled into the plugin install. The agent reaches for them via Glob/Grep when answering.
  • A slim always-loaded preamble (agents/sui-pilot-agent.md, ~2.9 KB) — a topic-routing table that tells the agent which corpus has what. No precomputed file index.
  • Five procedural slash skills (/move-code-quality, /move-code-review, /move-pr-review, /move-tests, /oz-math), each a self-contained Markdown skill body the agent loads when invoked.
  • An MCP server (move-lsp) that bridges move-analyzer (the official Move LSP) into Claude Code, exposing move_diagnostics, move_hover, move_completions, move_goto_definition.

That's the entire surface. No matcher pipeline, no hooks, no precomputed indexes — the agent reaches for docs when it needs them, and you type a slash command when you want a procedural skill.

3. The experiment: adopting then rolling back a sibling-plugin pattern

3.1 The hypothesis

Several Claude Code plugins have converged on a particular architecture pattern for skill injection: a hook-based matcher that watches tool calls and prompts, scores them against per-skill signals (path globs, bash patterns, import patterns, prompt phrases), and injects the top-scoring skill bodies plus extracted chunks from a curated relational graph (vercel.md for vercel-plugin, etc.). Vercel's own Claude Code plugin is a serious reference implementation.

The hypothesis: that pattern would make sui-pilot meaningfully better. So we adopted it on a branch (feat/v2-graph-port), drawing heavily on vercel-plugin's implementation as the reference. Three layers landed:

  1. Slim always-loaded preamble — replaced the legacy 19.4 KB pipe-delimited file index with a 2.9 KB topic-routing table.
  2. Ecosystem graph — a hand-curated sui.md (445 lines, 13 sections) with cross-references between concepts and skill markers, modelled on vercel.md.
  3. Hooks pipeline — a SessionStart profiler, a PreToolUse path/bash/import matcher, a UserPromptSubmit prompt scorer, byte budgets, dedup, a /sui-pilot-doctor health check, a build step that precompiled skill frontmatter into a manifest. ~7,500 LOC of TypeScript source / ~15 K LOC including compiled JS.

This is the architecture that vercel-plugin uses. The bet was that it would transfer.

3.2 The check

We built a 15-task A/B comparison suite covering common Sui/Move tasks — Move 2024 syntax migrations, OTW pattern, dynamic object fields, transfer policies, randomness, Walrus blob anchoring, Seal access policies, SDK 2.0 migration. Each task starts from a fixture (a minimal Move package or TS source tree), the model is given a prompt to make a specific change, and a grader checks whether the post-state satisfies pass criteria.

We ran the suite twice with the v1 and v2-with-matcher arms:

2026-04-30 baselinev1 (main, pipe-delimited preamble)v2 (with borrowed matcher pipeline)
Literal grader pass rate15/1513/15
Functional (after adjudicating 4 grader artefacts)13/1513/15

Tie on functional outcome. The literal gap was four grader artefacts in opposite directions — TODO-comment false positives on v1, idiomatic-form false negatives on v2. After adjudication: parity. The eval also flagged a verbose-TODO regression on v2: tasks 09 / 14 / 15 showed v2 deferring implementation to TODO comments more often, plausibly because the matcher over-injected reading material and the agent satisficed on implementation.

That run killed the confidence that the borrowed pattern was earning its complexity. The matcher solves prompt-disambiguation for plugins with many competing skills (vercel-plugin ships ~19); sui-pilot ships 5 explicitly-named slash skills, so the disambiguation problem doesn't exist here. The pattern is well-designed for the problem it solves; sui-pilot just doesn't have that problem.

3.3 The rollback

We rolled back the borrowed parts in a single commit:

  • The entire hooks/ tree (matcher, profiler, lexical index, manifest builder, doctor) — ~15 K LOC including the compiled .mjs bundles.
  • The sui.md ecosystem graph and sui-session.md nudge — both consumed only by the hooks.
  • The generated skill-manifest.json and the script that built it.
  • The doctor + verify scripts + /sui-pilot-doctor slash command.
  • Matcher metadata frontmatter from all 5 skills.

~17,389 LOC net deletion across 57 files. We kept the slim preamble (which the eval validated as a clean win on always-loaded bytes regardless of the matcher) and the eval harness itself (the load-bearing piece — see §5).

Then we re-ran the suite:

2026-05-11 post-rollback baselinev1 (main)v2 (rollback)
Literal grader pass rate14/1514/15

Dead heat. The trimmed v2 produced bit-identical outputs to v1 on every passing task. The precut verbose-TODO regression resolved without the matcher.

Empirical validation of the rollback: ~17 K LOC of borrowed complexity removed with zero observable quality cost. The simpler shape ships.

4. The eval framework

The eval harness in evals/ was built ad-hoc to validate the rollback, but generalises to any A/B comparison between Claude Code plugin shapes targeting Sui Move development. The rest of this section is its operator-and-design narrative.

4.1 Architecture

4.2 Task categories

The 27-task suite spans 5 categories, each surfacing different architectural pressure:

CategoryNWhat it stresses
tier-115 Single-file criterion-targeted edits — Move 2024 syntax (file-level module, vector method form, do! macro), OTW, dynamic object fields, transfer policies, randomness, Walrus blob anchoring, Seal encrypt, SDK 2.0 client migration, enum + match. The "easy" suite: both architectures converge here.
multi-file4 Renames across modules, monolith → state/entry split, multi-file SDK migration, DOF + accessor + test. Strains the agent's ability to maintain coherent reasoning across files.
ambiguous3 "Make this safer", "add a withdraw fee", "clean up naming" — many valid solutions. Scored via a 4-criterion LLM rubric (correctness / idiomaticity / safety / brevity, each 1–5, threshold ≥16).
stale-training4 Post-cutoff or rarely-trained Sui APIs: sui::derived_object, the GraphQL client (@mysten/sui/graphql), Coin Registry migration, sui::display::Display<T>. The correct answer requires reading the bundled docs; training memory will produce the wrong API shape.
token-pressure1 A ~1.5 KB structured spec with 6 acceptance bullets and 4 edge-case aborts. Designed to strain the model's reasoning budget on a single hard task.

Authoring guidance: pick a category that surfaces the comparison you care about, not whatever's easiest to write. Single-file substring-matched tasks (the tier-1 shape) are usually the wrong tool — both architectures converge fast and the eval tells you nothing.

4.3 Pass-criteria types

The schema is additive — old tasks stay valid, new tasks opt into richer checks:

FieldPurpose
containsStringThe target file must contain the literal substring after the model's edit. Matches inside Move/TS comments are rejected.
containsRegexPerl-compatible regex variant. Supersedes containsString when both present. Use when accepting multiple valid forms (e.g. accepting both coin::create_currency(otw, and coin::create_currency<DEMO>(otw,).
doesNotContainString / doesNotContainRegexNegative checks — the substring/pattern must not appear (excluding comments). The regex variant lets you use word boundaries (\bTransactionBlock\b) so legitimate longer identifiers like dryRunTransactionBlock survive.
alsoContainsString / alsoContainsRegexSecond positive check, same semantics.
additionalFilesArray of {file, ...checks} for multi-file tasks. Each entry checked independently.
compileAfter: trueRuns sui move build --build-env mainnet on the post-state. Pass requires exit 0. Skipped (not failed) when sui isn't on PATH.
rubricReplaces literal checks with an LLM-graded scorecard for ambiguous tasks: 4 criteria scored 1–5 each, gated on sum >= passThreshold.

4.4 Scoring

Each per-task artefact (diff, model text, token usage, optional compile-gate exit) feeds into a final claude -p turn that applies the pass criteria in order:

  1. Target file present in diff. If the task's passCriteria.file doesn't appear as a modified file, FAIL with "no edit to target file".
  2. Content criteria on the target file's post-state: regex/string positive checks, regex/string negative checks, comment-exclusion applied.
  3. Additional-file checks for multi-file tasks.
  4. Compile gate: if compileAfter: true, the build must have exited 0.
  5. Rubric: for ambiguous tasks, the scorer rates each rubric criterion 1–5 and gates on the sum.

The first failing criterion is the reason; otherwise PASS. Output is a self-contained HTML report (score.html) with headline pass rates, per-task tables, a "differentiators" section listing tasks where versions disagree (the architecturally interesting cases), a per-category breakdown, token economics from the aggregated tokens.csv, and a plain-English verdict paragraph.

4.5 Why no "no-plugin" control arm?

Considered, dropped. claude --bare refuses OAuth/keychain auth and requires ANTHROPIC_API_KEY (Console-API billed, separate from Claude Max plans), and the disable-the-plugin alternative needs a plugin-state restore dance that's too fragile for the marginal information it produces. The plugin's qualitative value over no-plugin is already obvious; the interesting comparison is between plugin shapes (matcher pipeline vs slim, pipe-delimited preamble vs topic-routing table, etc.).

5.4 Verdict

Is the rollback safe? Yes, conclusively. Quality parity across every difficulty tier; byte-identical outputs on the most ambiguous fixture.

Does the slim shape deliver productivity gain over the pipe-delimited preamble? No measurable gain on this fixture set. The slim preamble trades smaller cached preamble (−8 % cache_create) for more per-task doc reading (+26 % cache_read, +17 % output tokens) — net ~10 % more expensive on the public Anthropic rate card. The "slim preamble saves tokens" claim is true in isolation but doc-first per-task reads more than compensate on fresh claude -p invocations.

So why ship the slim shape? Maintenance complexity: −17 K LOC of borrowed-but-unused machinery removed. A smaller surface area to maintain, debug, and explain, for no quality cost.

6. What we learned

Lesson 1

Borrowed architecture patterns are hypotheses, not conclusions. Adopting the matcher pipeline was a defensible bet — the pattern is well-engineered, broadly used, validated by a serious team. But the value any pattern delivers depends on the host plugin's shape, and sui-pilot's shape (5 explicitly-named slash skills) doesn't match the shape the matcher solves (many overlapping skills, ambiguous prompts). When copying a pattern from a sibling plugin, name the problem the pattern solves first; if your plugin doesn't have that problem, the pattern is overhead.

Lesson 2

Eval before refactor. The 15-task suite caught what review wouldn't have. The rollback was justified by data, not vibes. Significant architectural decisions for plugins like this should land an eval baseline first — even a small one. Each iteration of the eval (precut, postcut, denser) revealed something the previous one couldn't have.

Lesson 3

Substring grading is noisy, and the grader is the living artifact. Half the work on this project was iteratively tightening the grader as the suite surfaced new false-positive and false-negative classes: TODO-comment substring matches, idiomatic Move 2024 forms the grader didn't recognise, RPC-method-name false positives (the dryRunTransactionBlock bug), un-passed --build-env flags. Plan for the grader to be edited every run for the first half-dozen runs. Comment-exclusion, regex variants, and word-boundary patterns are not optional features.

7. What's next

  • Grader sophistication. Current grader is substring + regex + comment exclusion + compile + LLM rubric. Next step: per-task sui move test runs to validate generated tests behave correctly, not just compile. For TS tasks, tsc --noEmit against the post-state would catch type errors.
  • A denser, harder suite. The Tier-2 fixtures still under-stress the model. Real-world Move repos with planted issues, multi-package workspaces, and ambiguous specs that require trade-off decisions would surface architectural deltas the current suite misses.
  • Cross-plugin eval pattern. This harness is sui-pilot-centric (Move/TS fixtures, sui-pilot's bundled docs assumed). Extracting it into a reusable Claude Code eval pattern (akin to the corpus-QA skill pattern) would let hypothetical walrus-pilot / seal-pilot projects use the same scaffolding.
  • CI integration. Currently on-demand. Wiring score.html's aggregate pass-rate into CI as a regression gate (block merges where v2 pass-rate < current main on a frozen subset) would prevent silent skill degradation. Cost-aware design needed — running 54 claude -p per PR is non-trivial.
  • Cross-model comparison. The harness uses whatever model claude -p defaults to. Wiring --model through the runner would let you A/B the same task set across Opus / Sonnet / Haiku and see which gives the most bang per token on Sui work.

8. For other Sui-related skills

The harness is intentionally generic. To evaluate any Sui-related Claude Code customization, replace the comparison axis:

  • Comparing two prompts for the same skill — point --v1-ref and --v2-ref at two branches that differ only in skills/your-skill/SKILL.md.
  • Comparing two models — wire --model into the runner's claude_cmd array and run the same suite under both. Then compare quality, tokens, and latency.
  • Comparing a new MCP integration — branch the plugin with the new MCP wired in, run the same suite.
  • Comparing two bundled-doc strategies — branch the plugin with different doc subsets, run the same suite, see which docs the agent actually reaches for and whether the answers differ.

Pick fixtures that target your skill's claimed value. A code-review skill should be evaluated against real-world Move code with planted issues, scored via the LLM rubric on whether the review caught them. A test-generation skill should be evaluated by running sui move test on the generated tests and checking they're meaningful (test the public API, not just the happy path). A migration skill should be evaluated with containsRegex for the new API form and doesNotContainRegex with word boundaries for the deprecated one.

The harness doesn't care what your skill does — it cares that there's a measurable post-state. Make the post-state observable, and you can measure it.
